# blog-expression-review

블로그 MDX 파일의 **의미 판단이 필요한 표현 규칙**을 검토하고 자동 수정합니다.
validator 가 처리한 확정적 패턴 다음에 오는 단계예요. validator 가 통과시킨
파일을 받아 글 전체를 통독하면서 어조와 표현의 자연스러움을 평가합니다.

**이 skill 은 SHARED.md 의 다음 섹션을 전제**합니다. 시작 시 Read 로 주입:

- `§BLOG-VOICE` — 어미 비율, 톤 예시
- `§RULE-FORBIDDEN-PATTERNS` — 병렬 삼단, 과장 형용사, 메타 문장
- `§RULE-RHYTHM` — 호흡 단조로움, 자기 목소리
- `§UI-USER-CHOICE` — 사용자 선택지 반환 형식 (오케스트레이터 경유 시)

다른 규칙 (em-dash, 콜론, `**`, bare 리스트, 영어 인용) 은 validator 가 이미
처리했으므로 **검사하지 않음**.

## 검토 흐름

validator 와 달리 expression-review 는 **Phase 가 아니라 통독 + 항목별 분석** 구조예요.
글 전체를 한 번에 머리에 띄워두고 6가지 항목을 동시에 추적해요.

```
Step 1: 파일 전체 Read (통독 모드)
Step 2: 6가지 항목 동시 분석
  - A3 병렬 삼단
  - A4 과장 형용사
  - A5 메타 문장
  - B1 어미 비율
  - B2 호흡
  - B3 자기 목소리
Step 3: 자동 수정 가능한 것 처리 (A5, B1, B2)
Step 4: 사용자 확인 필요한 것 구조화 (A3, A4, B3)
Step 5: 리포트 생성
```

---

## Step 1: 파일 전체 Read

대상 파일의 **frontmatter 부터 References 까지 전체** 를 Read.

```bash
cat <file>
```

또는 Read 툴로 limit 없이 전체 로드. expression-review 는 통독이 핵심이라 부분
Read 는 의미 없어요.

**시리즈 모드** 일 때: 각 편을 순서대로 통독. 편 사이 어조 일관성도 함께 봄.

---

## Step 2: 6가지 항목 동시 분석

글을 한 번 통독하면서 아래 6개를 머리에 띄워두고 추적해요. 항목별로 따로 통독하면
토큰 낭비고 일관성도 떨어져요.

### A3 — 병렬 삼단 (§RULE-PARALLEL-THREE)

**검사 방법**:

1. 본문에서 "X, Y, Z" 또는 "X 하고 Y 하고 Z" 같은 3단 병렬 패턴 찾기
2. 각 매칭에 대해:
   - **자연스러운 1회** 인가? — 통과
   - **섹션마다 반복** 되나? — 위반
3. 한 글에 2건 이상이면 **위반** 으로 분류

**판단 기준**:

- 형용사 3단 ("빠르고, 안전하고, 쉽습니다") — 가장 의심스러움
- 명사 3단 ("X, Y, Z 를 다룹니다") — 맥락에 따라 자연스러울 수 있음
- 동사 3단 ("읽고, 쓰고, 실행합니다") — 의심스러움
- 같은 구조의 반복 — 가장 강한 신호

**예시**:

```
❌ "이 함수는 빠르고, 안전하고, 직관적입니다."
   (형용사 3단, 마케팅 톤)

✅ "함수의 입력은 ID, 이름, 나이입니다."
   (명사 3단, 사실 나열)

❌ "처음엔 빠르고, 다음엔 정확하고, 마지막엔 안정적이에요."
   (구조 반복, 기계적)
```

**자동 수정**: 불가. 문장 재작성 필요.

**처리**: 사용자 확인 필요 카테고리로 분류. 위치와 원문, 수정 제안 2~3개를 구조화.

### A4 — 과장 형용사 (§RULE-HYPE)

**검사 방법**:

1. SHARED.md §RULE-HYPE 의 금지 단어 리스트 로드
2. 본문에서 각 단어 grep
3. 매칭마다 **문맥 판단**:
   - "강력한 타입 시스템" 같은 관용구? — 통과 (또는 경고)
   - "완벽한 솔루션" 같은 마케팅? — 위반
4. 위반 항목마다 **구체 표현 대체안** 생성

**판단 기준**:

- 단어 자체가 아니라 **문맥에서 구체성을 제거하는 칭찬** 인지
- 대체할 구체 표현 (수치, 동작, 예시) 이 있는가?

**예시**:

```
❌ "강력한 성능을 제공합니다"
   대체: "1000건 / 초 처리 가능합니다" 또는 "메모리 사용량이 절반"

✅ "강력한 타입 시스템" (관용구, 일부 통과)
   하지만 가능하면 "엄격한 타입 추론" 으로 대체

❌ "완벽한 해결책"
   대체: 구체 사례 또는 "이 문제의 한 가지 답"
```

**자동 수정**: 부분적으로 가능. 단어 자체는 자동 제거 가능하지만 대체 표현은 맥락 필요.

**처리**:

- 명백한 마케팅 표현 (`완벽한`, `놀라운` 등) 은 자동 제거 후 보고
- 대체 표현이 명확한 경우 (`강력한 성능` → 구체 수치 권장) 는 사용자 확인

### A5 — 메타 문장 (§RULE-META)

**검사 방법**:

1. SHARED.md §RULE-META 의 금지 패턴 리스트 로드
2. 본문에서 패턴 grep
3. 변형도 함께 (예: "다뤄봅니다", "다뤄볼게요", "다뤄볼까요")
4. 각 매칭 위치 판단:
   - 글 시작부 (오프닝) — 가장 흔한 위치
   - 섹션 도입부 — 종종
   - 마무리 — "결론적으로" 같은 패턴

**예시**:

```
❌ 오프닝: "이번 글에서는 React useMemo 의 동작 원리를 다뤄보겠습니다."
   처리: 통째로 삭제. 다음 줄로 시작.

❌ 마무리: "결론적으로 말씀드리면 useMemo 는 신중히 사용해야 합니다."
   처리: "결론적으로 말씀드리면" 삭제. "useMemo 는 신중히..." 로 시작.

❌ 섹션 도입: "이제 useMemo 의 의존성 배열을 살펴보겠습니다."
   처리: 통째로 삭제.
```

**예외**:

- 시리즈 중간 편의 "다음 편 예고" → 메타 아님, 통과
- "이 글에서는" 으로 시작하는 도입이 한 번만 나오고 절제됐다면 통과

**자동 수정**: **가능**. 메타 문장은 통째로 삭제하면 글이 자연스러워짐.

**처리**: 자동 삭제 후 수정 로그에 기록.

### B1 — 어미 비율 (§BLOG-VOICE)

**검사 방법**:

1. 본문 전체에서 종결 어미 카운팅:
   - 구어 존댓말: `~에요`, `~예요`, `~죠`, `~거든요`, `~고요`, `~어요`, `~네요`
   - 격식체: `~입니다`, `~합니다`
   - 반말 문어체: `~다`, `~이다`, `~한다`, `~된다`, `~있다`
2. 비율 계산:
   - 구어 존댓말 70% 이상 → 통과
   - 격식체 30% 초과 → 경고
   - 반말 문어체 1건이라도 → **블로커급 위반**
3. 추가 체크: **한 섹션** 이 통째로 격식체로 채워졌는지

**판단 기준**:

- 비율은 글 전체 기준
- 한 섹션이 격식체로 통째로 채워지면 그 섹션만 일부 구어로 변환 권장
- 반말 문어체는 무조건 전수 교체

**예시**:

```
❌ "useMemo 는 메모이제이션 함수다."
   → "useMemo 는 메모이제이션 함수예요."

❌ "이 함수는 의존성이 변경될 때만 재계산된다."
   → "이 함수는 의존성이 변경될 때만 재계산돼요."

⚠️ "useMemo 는 메모이제이션 함수입니다. 의존성 배열을 받습니다.
    값을 캐싱합니다. 성능 최적화에 사용됩니다."
   → 한 섹션이 통째로 격식체. 한두 곳을 구어로:
     "useMemo 는 메모이제이션 함수예요. 의존성 배열을 받죠..."
```

**자동 수정**: **가능**.

- 반말 문어체 → 구어 존댓말 1:1 매핑 (확정 변환)
- 격식체 일부를 구어로 (한 섹션 통째 격식체일 때)

**처리**:

- 반말 문어체 발견 시 자동 전수 교체 → 수정 로그 기록
- 격식체 비율 초과 시 자동 일부 변환 → 수정 로그 기록

### B2 — 호흡 (§RULE-RHYTHM-VARIED)

**검사 방법**:

1. 본문을 문단 단위로 나누기
2. 각 문단에서 문장 길이 측정 (한국어 문장은 마침표 기준)
3. 패턴 검출:
   - 같은 종결 어미가 3번 이상 연속 → 위반
   - 긴 문장 (60자 이상) 이 3개 이상 연속 → 위반
   - 짧은 문장만 (20자 미만) 5개 이상 연속 → 위반 (덜 흔함)

**판단 기준**:

- 문단 단위로 본다 (글 전체 평균은 의미 없음)
- 같은 어미 반복이 가장 명확한 위반
- 긴 문장 연속은 가독성 떨어뜨림

**예시**:

```
❌ "useMemo 는 메모이제이션 함수예요. 의존성 배열을 받아요.
    값을 캐싱해요. 성능 최적화에 도움이 돼요."
   (4문장 모두 ~요 종결, 단조)

✅ "useMemo 는 메모이제이션 함수예요. 의존성이 변경될 때만 재계산하죠.
    근데 모든 값에 useMemo 를 쓰면 오히려 느려질 수 있어요. 캐싱 자체에도
    비용이 있거든요."
   (어미 다양, 길이 다양)
```

**자동 수정**: 부분적으로 가능. 같은 어미 반복은 일부를 다른 어미로 교체 가능.

**처리**:

- 같은 어미 3회 이상 반복 → 자동 일부 교체 (1~2개를 다른 어미로)
- 긴 문장 연속 → 사용자 확인 (어디서 끊을지 판단 필요)

### B3 — 자기 목소리 (§RULE-SELF-VOICE)

**검사 방법**:

1. 본문에서 1인칭 표현 검색:
   - "저", "저는", "제가", "저도"
   - "처음엔", "그 때", "한번은"
   - "~싶었어요", "~생각했어요", "~겪었어요"
   - 혼잣말 패턴: "음", "그러게요", "근데"
2. 카운팅:
   - 글 길이 3000자 이상이면 1~2건 이상 필요
   - 글 길이 1500~3000자면 1건 이상 권장
   - 짧은 글 (1500자 미만) 은 없어도 통과
3. 위치 확인: 오프닝, 본문 중간, 마무리 중 어디든 OK

**판단 기준**:

- 1인칭 경험이 글의 친근함을 만듦
- 없으면 교과서 톤이 됨
- 너무 많으면 (5건 이상) 산만함

**예시**:

```
✅ "처음엔 useMemo 를 모든 곳에 썼어요. 결과는 오히려 느려졌죠."
   (1인칭 경험, 자연스러움)

✅ "저도 처음엔 의존성 배열을 빠뜨려서 무한 루프에 걸렸어요."
   (시행착오 공유)

❌ 글 전체에 1인칭 표현 0건 + 3000자 이상
   → "어딘가에 1~2줄 추가하면 좋아요"
```

**자동 수정**: **불가**. 사용자가 직접 써야 함.

**처리**: 사용자 확인 필요 카테고리. "어느 자리에 추가하면 좋을지" 제안만 줌.

## Step 3: 자동 수정 가능한 것 처리

A5 메타 문장, B1 반말 문어체, B1 격식체 비율, B2 어미 반복은 **확정적으로 수정 가능**.
이 항목들은 직접 Edit 으로 수정.

### Edit 원칙

1. **수정 전 백업 불필요** — validator 가 이미 한 번 처리한 파일이고, expression-review
   의 수정도 작은 단위라 백업 부담 큼. 단, 실수 시 사용자가 git 으로 되돌릴 수 있어야 함.
2. **수정 후 재검증** — 같은 항목을 한 번 더 검사해서 수정이 잘 됐는지 확인
3. **수정 로그 기록** — 리포트에 어떤 수정을 했는지 명시

### 자동 수정 예시

**A5 메타 문장 자동 삭제**:

```
[A5 메타 문장] L5 오프닝
원문: "이번 글에서는 useMemo 의 동작 원리를 다뤄보겠습니다."
처리: 라인 통째로 삭제
이후: 다음 단락으로 자연스럽게 연결되는지 확인
```

**B1 반말 문어체 자동 교체**:

```
[B1 반말 → 구어 존댓말] 5건 자동 교체
- L23 "useMemo 는 메모이제이션 함수다." → "useMemo 는 메모이제이션 함수예요."
- L34 "재계산된다." → "재계산돼요."
- L45 "사용한다." → "사용해요."
- L67 "실행된다." → "실행돼요."
- L89 "있다." → "있어요."
```

매핑 테이블:

- `~다.` → `~예요.` (명사 + 다)
- `~한다.` → `~해요.`
- `~된다.` → `~돼요.`
- `~있다.` → `~있어요.`
- `~이다.` → `~이에요.`

**주의**: 자동 매핑은 단순 케이스만. 복잡한 문장 (`있을 수 있다.`, `~할 수 있다.` 등) 은
사용자 확인 카테고리로.

**B1 격식체 일부를 구어로**:

한 섹션이 통째로 격식체일 때, **마지막 두세 문장만** 구어로 변환:

```
[B1 격식체 비율 조정] 섹션 "useMemo 의 동작" — 8문장 중 모두 격식체
처리: 마지막 2문장을 구어로 변환
- L78 "성능 최적화에 사용됩니다." → "성능 최적화에 써요."
- L80 "신중하게 사용해야 합니다." → "신중하게 써야 해요."
```

**B2 어미 반복 일부 교체**:

같은 어미 4번 연속이면 가운데 1~2개를 다른 어미로:

```
[B2 어미 반복] L34~L40 "~해요" 4회 연속
처리: L36, L38 의 "~해요" 를 다른 형태로
- L36 "재계산해요." → "재계산하죠."
- L38 "캐싱해요." → "캐싱하거든요."
```

## Step 4: 사용자 확인 필요 항목 구조화

A3 병렬 삼단, A4 과장 형용사 (대체 표현 필요한 경우), B2 긴 문장 연속, B3 자기 목소리는
**오케스트레이터 경유로 사용자 확인** 필요.

§UI-USER-CHOICE 규칙에 따라, expression-review 는 **사용자와 직접 대화하지 않고**
구조화된 데이터를 오케스트레이터로 반환. 오케스트레이터가 AskUserQuestion 호출.

### 반환 형식

```markdown
## 사용자 확인 필요 항목

수정 제안:

1. "이 함수는 빠르고 안전해요. 게다가 직관적이죠."
2. "이 함수는 빠르고 직관적이에요. 안전성도 검증됐고요."

기본값: 제안 1

### 항목 2: [A4 과장 형용사 - 대체 표현] L78

원문:
"강력한 성능을 제공합니다."

문제: "강력한" 이 마케팅 톤. 구체 표현으로 대체 권장.

수정 제안:

2. "메모리 사용량이 절반으로 줄어요." (비교)

제안:
"useMemo 는 의존성 배열의 모든 항목이 변경되지 않았을 때만 캐싱된 값을

### 항목 4: [B3 자기 목소리 부재]

위치 후보:
```

## 시리즈 모드 추가 검사

`mode: "series"` 일 때, 각 편을 통독한 후 **편 사이 어조 일관성** 도 검사:

- 한 편이 다른 편보다 격식체가 많거나 적은가?
- 한 편에만 자기 목소리가 빠져 있는가?
- 한 편의 호흡이 다른 편과 다른가?

이건 자동 수정하지 않고 **시리즈 총평** 으로 보고:

```markdown
## 시리즈 어조 총평

3편 검토 결과:

- 1편 (use-memo-basics): 구어 비율 75%, 자기 목소리 2건 ✅
- 2편 (use-memo-deps): 구어 비율 60% ⚠️ (3편 중 가장 격식체)
- 3편 (use-memo-pitfalls): 구어 비율 78%, 자기 목소리 3건 ✅

권장: 2편의 격식체 비율을 다른 편 수준으로 낮추기. 자동 처리는 해뒀으나
어조 균형이 1, 3편과 약간 다름. 전체 통독해보고 어색하면 추가 조정.
```

## 제약

- **validator 가 처리한 영역은 검사하지 않음**. em-dash, 콜론, JSX prop `**`,
  bare 리스트, 영어 인용 풀이는 전부 validator 담당.
- **글의 의미를 바꾸는 수정 금지**. 예를 들어 "useMemo 는 빠릅니다" 를 "useMemo 는
  느립니다" 로 바꾸지 않음. 표현만 바꿈.
- **새 문장 창작 금지**. 자기 목소리 (B3) 추가는 사용자가 직접. expression-review 는
  위치만 제안.
- **반말 문어체 1:1 매핑만**. 복잡한 어미 변환 (`~할 수 있다` → `~할 수 있어요`)
  은 사용자 확인 필요.
- **시리즈는 편 단위로 통독**. 한 번에 여러 편 동시 분석 금지 (메모리/판단 부담).
- **SHARED.md 규칙을 SKILL.md 에 복사하지 마세요**. Read 로 참조만.
