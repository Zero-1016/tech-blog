# blog-coherence-review

블로그 MDX 파일의 **논리 완결성**을 검토합니다. 표면 패턴 (em-dash, 콜론, 어미)
이 아니라 글의 **논지 구조** 를 봐요. validator 와 expression-review 가 통과한
파일을 받아 글 전체를 통독하면서 도입-결론 호응, 섹션 간 논리, 근거 충실도,
내부 모순을 평가합니다.

**자동 수정을 거의 하지 않습니다.** 논리 재작성은 글의 메시지 자체를 바꿀 수
있어서 사용자 확인이 필수예요. 신중하게 지적하고 사용자 판단에 맡겨요.

**이 skill 은 SHARED.md 의 다음 섹션을 전제**합니다:

- `§UI-USER-CHOICE` — 사용자 선택지 반환 형식

다른 SHARED.md 섹션은 검사 대상이 아니에요. coherence-review 는 글의 의미 구조만
봄.

## 신중함의 원칙

coherence-review 는 다른 리뷰어보다 신중해야 해요. 이유:

1. **주관이 들어감** — "이 결론이 도입과 호응하는가?" 는 해석의 여지가 큼
2. **글쓴이가 의도한 흐름** 을 LLM 이 잘못 읽을 수 있음
3. **자동 수정 불가** — 잘못 지적해도 사용자가 무시하면 그만이지만, 잘못 자동 수정하면 글이 망가짐

### 신중 원칙 1: 의심스러우면 통과

확정적인 결함만 지적. 애매하면 통과시키고 사용자가 직접 발견하도록.

**나쁜 예**:

- "이 섹션이 약간 어색하게 연결되는 것 같아요" → 너무 모호

**좋은 예**:

- "L23 에서 'A 가 B 보다 빠르다' 고 단정했는데 본문에 측정 근거나 출처가 없어요.
  L67 의 W3C 인용을 L23 옆에 배치하면 근거가 됩니다."

### 신중 원칙 2: 근거 명시

각 지적에 **"왜 그렇게 읽혔는지" 근거 한 줄** 필수. 근거 없는 지적 금지.

### 신중 원칙 3: 취향 태그

확정 위반이 아닌 의견 차원의 지적은 `[취향]` 태그를 붙여서 사용자가 선택적으로
반영할 수 있게 해요.

```
[E2 섹션 연결] [취향] L45
섹션 3 → 섹션 4 전환이 살짝 갑작스러워요. 두 섹션이 같은 개념의 다른 측면을
다루는데, 연결 문장 한 줄 (예: "이 동작의 반대 케이스도 있어요") 이 있으면 더
자연스러워요. 하지만 현재도 잘못된 건 아니에요.
```

### 신중 원칙 4: 글쓴이 의도 추정

LLM 이 모르는 글쓴이의 의도가 있을 수 있어요. 예를 들어:

- 일부러 결론을 열어둠 (독자 사고 유도)
- 일부러 모순처럼 보이는 두 케이스를 나란히 둠 (대조 강조)
- 일부러 근거 없이 단정함 (글쓴이의 경험적 단언, 출처 불필요)

이런 의도를 침해하지 않도록, 지적할 때 **"의도가 다르면 무시해주세요"** 같은
여지를 둬요.


## Step 1: 파일 전체 Read

대상 파일의 frontmatter 부터 References 까지 전체 통독.

```bash
cat <file>
```

또는 Read 툴로 limit 없이.

**시리즈 모드** 일 때: 각 편을 순서대로 통독. 편마다 논지 구조를 메모하고, 마지막에
시리즈 통합 분석.

---

## Step 2: 글의 논지 구조 추출

통독하면서 다음을 머리에 그려요. 명시적으로 출력하지 않아도 되지만, 나중 단계에서
참조할 수 있게 정리.

### 추출할 것

1. **오프닝의 문제 제기**: 글이 시작하는 자리에서 어떤 문제/질문/장면을 제기했나?
2. **핵심 메시지**: 글이 결국 말하려는 한 줄
3. **섹션별 역할**: 각 H2 가 어떤 논지를 담당하나? (정의 / 원인 / 해결 / 예시 / 주의 등)
4. **결론의 마무리 지점**: 마지막 H2 또는 마무리 단락이 어디서 끝나나?
5. **주요 단정**: "X 는 Y 다" 식의 확정적 주장들 (E3 검사 대상)
6. **권장 / 비권장**: "X 를 권장합니다", "Y 는 피하세요" 식의 권고 (E4 검사 대상)

### 메모 형식 (내부용)

```
오프닝 문제: useMemo 를 모든 곳에 쓰면 오히려 느려질 수 있다는 의문
핵심 메시지: useMemo 는 측정 후 사용해야 한다
섹션:
  1. useMemo 의 동작 (정의)
  2. 언제 도움이 되는가 (효과 사례)
  3. 언제 해롭나 (역효과 사례)
  4. 측정 방법 (실용 가이드)
  5. 마무리 (요약 + 권장)
결론 마무리: "신중하게 측정 후 사용하세요"
주요 단정:
  - L23: "useMemo 는 캐싱 비용이 있다"
  - L45: "의존성 배열 비교는 얕은 비교다"
  - L78: "성능 최적화는 측정이 우선이다"
권장:
  - "측정 후 사용" (L120, 결론)
  - "프로파일링 도구 사용" (L95)
비권장:
  - "모든 함수에 useMemo" (L34)
```

## Step 3: E1~E4 동시 평가

논지 구조를 머리에 띄워두고 4개 항목을 동시에 평가.

### E1 — 도입-결론 호응

**검사 방법**:

1. 오프닝의 문제 제기를 다시 확인
2. 마무리 단락이 그 문제를 (명시적 또는 암시적으로) 해소하는지 평가
3. 호응 여부 판정:
   - **호응 ✅**: 오프닝에서 제기한 X 가 마무리에서 답해짐
   - **부분 호응 ⚠️**: 답해지긴 했는데 명확하지 않음
   - **불일치 ❌**: 오프닝과 마무리가 다른 주제를 다룸

**예시**:

```
✅ 호응
오프닝: "useMemo 를 모든 곳에 쓰면 왜 느려질까?"
마무리: "useMemo 는 측정 후 신중히 사용하세요. 캐싱 비용이 있거든요."
→ 오프닝 의문에 마무리가 답함

❌ 불일치
오프닝: "useMemo 를 모든 곳에 쓰면 왜 느려질까?"
마무리: "결국 React 의 렌더링 최적화는 컴포넌트 설계가 더 중요해요."
→ 오프닝 의문이 해결되지 않고 다른 주제로 빠짐
```

**판정 기준**:

- 호응이 명확하면 통과
- 부분 호응이면 [취향] 태그로 지적
- 불일치면 확정 위반으로 지적

### E2 — 섹션 간 논리 연결

**검사 방법**:

1. H2 섹션 순서를 확인
2. 각 전환점에서 평가:
   - 앞 섹션이 뒷 섹션의 전제가 되는가?
   - 뒷 섹션이 앞 섹션의 답이 되는가?
   - 또는 앞 섹션의 다른 측면을 다루는가?
3. 논리적 점프 또는 어색한 연결 발견 시 지적

**판정 기준**:

- "왜 이 섹션이 지금 오는가?" 가 자연스러우면 통과
- 갑작스러운 주제 전환이면 지적
- 앞 섹션 내용이 뒷 섹션에서 딴소리처럼 다뤄지면 지적

**예시**:

```
✅ 자연스러운 전환
섹션 1: useMemo 의 동작 (정의)
섹션 2: 언제 도움이 되는가 (정의 → 효과)
섹션 3: 언제 해롭나 (효과의 반대)
→ 자연스러운 흐름

❌ 논리 점프
섹션 1: useMemo 의 동작
섹션 2: React 18 의 동시성 모드
→ useMemo 와 동시성 모드의 관계가 본문에 없으면 갑작스러움
```

**False Positive 주의**: 글쓴이가 의도적으로 대조나 우회 구조를 쓸 수 있어요. 명백한
점프만 지적.

### E3 — 근거 없는 단정

**검사 방법**:

1. Step 2 에서 추출한 "주요 단정" 리스트 확인
2. 각 단정에 대해 본문에 근거가 있는지:
   - 수치 / 측정 결과
   - 출처 인용 (`<Cite>` 태그 또는 References 의 항목)
   - 코드 예시
   - 글쓴이의 경험 ("저도 측정해봤는데") — 약하지만 근거
3. 근거 없는 단정이면 지적

**판정 기준**:

- 일반론 ("React 는 단방향 데이터 흐름이다") 은 근거 없어도 OK — 상식
- 비교 단정 ("A 가 B 보다 빠르다") 은 근거 필수
- 절대 단정 ("X 는 항상 Y 다") 은 근거 필수
- 권고 단정 ("X 를 사용하세요") 은 이유 필수

**예시**:

```
❌ 근거 없는 비교
"useMemo 는 useCallback 보다 무겁습니다."
→ 본문에 측정 근거나 출처 없음 → 지적

✅ 근거 있는 비교
"useMemo 는 useCallback 보다 무겁습니다.<Cite id="react-perf" />"
→ References 에 근거 있음 → 통과

⚠️ 약한 근거
"제가 프로파일링해봤을 때 useMemo 가 약 5% 느렸어요."
→ 글쓴이 경험, 측정 자체는 있지만 재현 어려움 → 통과 (1인칭 경험)
```

**판정 기준**:

- `<Cite>` 또는 References 매칭이 있으면 통과
- 글쓴이 1인칭 경험이면 통과
- 그 외 단정은 지적

### E4 — 내부 모순

**검사 방법**:

1. Step 2 에서 추출한 "권장 / 비권장" 리스트 확인
2. 같은 대상에 대해 권장과 비권장이 동시에 나오는지 검사
3. 모순처럼 보이면 **조건 차이** 가 본문에 명시되어 있는지 확인

**판정 기준**:

- 무조건 모순이면 지적
- 조건 차이가 있으면 통과 (예: "작은 컴포넌트엔 useMemo 불필요, 큰 리스트엔 필요")
- 조건 차이가 모호하면 [취향] 태그로 "조건을 명시하면 좋다" 지적

**예시**:

```
❌ 명백한 모순
L23: "useMemo 는 모든 컴포넌트에 사용하세요."
L78: "useMemo 는 거의 사용하지 마세요."
→ 조건 차이 없이 정반대 권고 → 지적

✅ 조건이 명시된 대조
L23: "리스트 렌더링이 무거우면 useMemo 사용을 고려하세요."
L78: "단순한 컴포넌트에는 useMemo 가 오히려 오버헤드입니다."
→ 조건 (리스트 vs 단순) 이 명시됨 → 통과

⚠️ 조건이 모호한 대조
L23: "useMemo 를 사용하세요."
L78: "useMemo 를 피하세요."
→ 조건 명시 안 됨 → [취향] 으로 "조건을 명시하면 좋다" 지적
```

**시리즈에서의 E4**:

시리즈는 E4 가 가장 잘 발동해요. 1편에서 권장한 패턴이 3편에서 부정될 수 있거든요.
시리즈 모드에서는 **각 편의 권장/비권장 리스트를 모아서** 비교.

## Step 4: 시리즈 모드 추가 검사

`mode: "series"` 일 때만.

각 편을 통독한 후 시리즈 전체를 다시 한 번 볼 때 검사:

### S1 — 시리즈 전체 도입-결론 호응

- 1편 오프닝에서 제기한 문제가 마지막 편 마무리에서 해결되는가?
- 시리즈 전체가 하나의 큰 논지로 이어지는가?

### S2 — 편 사이 논리 연결

- 1편 → 2편 전환이 자연스러운가?
- 2편이 1편을 전제로 하는가?
- 마지막 편의 회고가 모든 편을 통합하는가?

### S3 — 편 사이 모순

- 1편에서 권장한 X 가 3편에서 부정되지 않는가?
- 동일 개념의 정의가 편마다 일치하는가?
- 코드 예시 스타일이 편마다 일관적인가?

### S4 — 편 분량 균형

- 한 편이 다른 편보다 압도적으로 길거나 짧지 않은가?
- 편마다 다루는 깊이가 비슷한가?

### 시리즈 총평 형식

```markdown
## 시리즈 총평

3편 검토 완료.

### 호응 (S1)

1편 오프닝 "왜 useMemo 가 느릴 수 있나?" → 3편 마무리 "측정 후 사용하세요"
✅ 호응

### 편 사이 논리 (S2)

1편 (정의) → 2편 (역효과 사례) → 3편 (측정 가이드)
✅ 자연스러운 흐름

### 편 사이 모순 (S3)

- 2편 L45 "useMemo 를 절대 쓰지 마세요" vs 3편 L67 "측정 후 사용하세요"
  → ⚠️ 2편의 절대 부정과 3편의 조건부 권고가 약간 어긋남. 2편의 톤을
  "기본적으로 피하세요" 정도로 완화하면 더 부드러움.

### 편 분량 (S4)

1편 2200자, 2편 2400자, 3편 1800자
✅ 균형
```

## Step 5: 사용자 확인 항목 구조화

E1~E4 (그리고 시리즈 S1~S4) 에서 발견한 항목을 정리. 거의 모든 항목이 사용자 확인
필요 카테고리예요 (자동 수정 안 함).

### 분류

- **확정 위반** — 명백한 결함, 신중하게 검토해서 사용자에게 강조
- **취향 차원** — `[취향]` 태그, 사용자가 선택적 반영
- **시리즈 통합** — 시리즈 모드에서만 발생, 편 단위가 아닌 시리즈 단위 지적

### 반환 형식

```markdown
## 사용자 확인 필요 항목

### 항목 1: [E1 도입-결론 불일치]

문제:

- 오프닝 (L8): "useMemo 를 모든 곳에 쓰면 왜 느려질까?"
- 마무리 (L145): "결국 React 의 렌더링 최적화는 컴포넌트 설계가 더 중요해요."

이렇게 읽힌 근거:
오프닝의 의문 ("왜 느려질까") 에 대한 답이 마무리에 없고, 다른 주제 (컴포넌트 설계) 로
넘어갔어요.

수정 옵션:

A) 마무리에 오프닝 의문에 대한 답을 추가
"useMemo 는 측정 후 사용하세요. 캐싱 비용이 있어서 무분별하게 쓰면 느려지거든요.
더 근본적으로는 컴포넌트 설계가 우선이고요."

B) 오프닝을 마무리 주제에 맞게 수정
"React 의 렌더링 최적화, 어디서부터 시작해야 할까?"

C) 유지 (글쓴이가 의도한 흐름이라면)

### 항목 2: [E3 근거 없는 단정] [취향]

위치: L67

문제:
"useMemo 는 useCallback 보다 약 30% 무겁습니다."

이렇게 읽힌 근거:
30% 라는 수치를 단정했는데 본문/References 에 측정 근거나 출처가 없어요. 실측이거나
인용이면 출처 명시가 좋아요.

수정 옵션:

A) 출처 추가
"useMemo 는 useCallback 보다 약 30% 무겁습니다.<Cite id="react-perf-bench" />"
References 에 해당 출처 추가 필요

B) 수치 제거하고 일반화
"useMemo 는 useCallback 보다 다소 무겁습니다."

C) 글쓴이 측정이라면 명시
"제가 프로파일링했을 때 약 30% 차이가 났어요."

D) 유지

### 항목 3: [E4 내부 모순]

문제:

- L23: "리스트 렌더링은 useMemo 로 감싸세요."
- L78: "리스트 렌더링은 React.memo 가 더 효율적이에요."

이렇게 읽힌 근거:
같은 "리스트 렌더링" 에 대해 L23 은 useMemo 권장, L78 은 React.memo 권장. 두 도구의
역할이 다른데 본문에 차이가 명시되지 않아서 독자가 헷갈릴 수 있어요.

수정 옵션:

A) L23 과 L78 사이에 두 도구의 차이 명시
"useMemo 는 값을 캐싱하고, React.memo 는 컴포넌트 자체를 캐싱해요. 리스트 렌더링은
React.memo 가 더 자연스럽지만, 리스트 항목 안의 무거운 계산이 있으면 useMemo 도
함께 써요."

B) 두 권고를 하나로 통합
"리스트 렌더링은 React.memo 우선, 그 안의 무거운 계산이 있을 때만 useMemo 추가."

C) 유지 (의도한 대조라면)
```

## 제약

- **자동 수정 거의 없음**. 모든 의미 변경은 사용자 확인 필수.
- **새 문장 창작 금지**. coherence-review 가 임의로 결론이나 도입을 새로 쓰지 않음.
  사용자가 직접 입력하도록 유도.
- **표면 패턴은 검사 안 함**. em-dash, 콜론, `**`, 어미, 호흡 등은 validator 와
  expression-review 담당.
- **의심스러우면 통과**. 신중함 원칙 준수.
- **각 지적에 근거 한 줄 필수**. "이렇게 읽힌 근거" 명시.
- **글쓴이 의도 추정**. LLM 이 모르는 의도가 있을 수 있으므로 "의도가 다르면
  무시해주세요" 같은 여지를 둠.
- **시리즈 모드는 편 단위 통독 후 시리즈 통합 분석**. 한 번에 여러 편 동시 분석 금지.
