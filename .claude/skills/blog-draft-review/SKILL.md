# blog-draft-review

블로그 글 **기획안**을 검토하는 skill. validator/expression/coherence 와 달리
**작성 전** 검토라 입력이 MDX 파일이 아니라 기획안 마크다운이에요.

writer 가 헛수고 하지 않게 기획 단계에서 결함을 잡는 게 목적. 자료 부족, 복잡도
오판, 제목 약함, 빈약한 섹션 같은 걸 미리 발견.

**이 skill 은 SHARED.md 의 다음 섹션을 전제**합니다:

- `§SOURCE-PRIORITY` — 출처 우선순위 (D1 검사용)
- `§META-TITLE` — 제목 기준 (D3 검사용)
- `§META-DESCRIPTION` — 설명 기준 (D3 검사용)
- `§COMPLEXITY` — 복잡도 판단 기준 (D2 검사용)
- `§UI-USER-CHOICE` — 사용자 확인 시 (오케스트레이터 경유)

다른 SHARED.md 섹션 (§RULE-\*, §BLOG-VOICE 등) 은 검사 대상 아님 — 그건 글이 작성된
후 validator/expression/coherence 의 영역.

---

## 입력 계약

draft-review 는 거의 항상 **오케스트레이터 호출**이에요. 단독 실행은 기획안이
별도 파일로 존재하지 않아서 어색해요. 그래도 단독 실행 케이스를 위한 입력 형식도
정의.

### 오케스트레이터 호출 시

```
plan: <Phase 3 에서 작성된 기획안 마크다운 전문>
sources: <Phase 2 blog-research 결과 — 출처별 구조화된 데이터>
internal_link_candidates: <Phase 1 결과 — 내부 링크 후보 + 배치 힌트>
topic: <원본 주제 문자열>
mode: <"single" 또는 "series">
via: "orchestrator"
```

### 단독 실행 시 (`/blog-draft-review`)

- 사용자가 기획안 마크다운을 직접 입력
- 또는 파일 경로 (`.draft-plan.md` 같은 임시 파일)
- sources, internal_link_candidates 는 누락 가능 — 누락되면 해당 검사 항목만 스킵

## 신중함의 원칙

draft-review 는 **사용자가 GATE 1 로 가기 전 단계** 에 끼어드는 거예요. 너무 까다로우면
사용자가 매번 추가 작업해야 해서 짜증나요.

### 원칙 1: 블로커 / 경고 / 취향 분류

발견한 결함을 세 단계로:

- **블로커 ❌**: GATE 1 로 진행하면 안 되는 결함. 반드시 처리.
- **경고 ⚠️**: 알리지만 진행 가능. 사용자가 결정.
- **취향 💭**: 선택적 개선 제안. 무시해도 무방.

### 원칙 2: 진행 가능한 기획안인가만 판단

draft-review 의 핵심 질문: **"이 기획안으로 writer 가 좋은 글을 쓸 수 있을까?"**

너무 좋은 기획안일 필요 없음. 그냥 writer 가 헛수고 안 할 정도면 됨.

### 원칙 3: 자동 수정 vs 사용자 확인

자동 수정 가능 (오케스트레이터에게 액션 반환):

- D1 자료 부족 → blog-research 재호출 권장
- D2 복잡도 오판 → 단편 ↔ 시리즈 전환 권장
- D3 제목 후보 부족 → 후보 추가 생성 권장
- D5 내부 링크 부정확 → grep 재검색 권장

사용자 확인 필요:

- D4 섹션 구조 큰 변경 (재배치, 추가, 삭제)
- 주제 자체의 재해석
- 시리즈 ↔ 단편 전환의 큰 결정

## Step 1: 입력 파싱

오케스트레이터 호출이면 입력이 이미 구조화돼 있음. 그대로 사용.

단독 실행이면 사용자가 입력한 기획안 마크다운에서 다음을 추출:

- 모드 (단편/시리즈)
- 제목 후보들
- 설명 (50자 내외)
- 핵심 메시지
- 섹션 구조 (H2 + 역할 + 자료 매핑)
- 자료 요약
- 내부 링크 후보

추출이 어려우면 사용자에게 형식 안내 후 재입력 요청.

## Step 2: D1~D5 분석

### D1 — 자료 충분성

**검사 방법**:

1. 기획안의 각 섹션이 어떤 자료에 매핑돼 있는지 확인
2. 매핑된 자료의 출처 우선순위 확인 (§SOURCE-PRIORITY)
3. 섹션별 평가:
   - **빈약 ❌**: 매핑된 자료 0개
   - **부족 ⚠️**: 1순위 자료 없음, 2순위 이하만
   - **충분 ✅**: 1순위 자료 1개 이상

**전체 평가**:

- 1순위 0개 → **블로커** (writer 가 신뢰할 자료 없음)
- 1순위 1~2개 → **경고** (보강 권장)
- 1순위 3개 이상 → **통과**

**섹션별 평가**:

- 빈약한 섹션 0개 → 통과
- 빈약한 섹션 1개 → 경고
- 빈약한 섹션 2개 이상 → 블로커

**예시**:

```
[D1 자료 충분성]

전체:
- 1순위 자료: 2개 (MDN, W3C) ⚠️ 경고: 3개 권장

섹션별:
- 섹션 1 "정의": MDN 매핑 ✅
- 섹션 2 "동작 원리": W3C 매핑 ✅
- 섹션 3 "실전 사례": 매핑 자료 0개 ❌ 빈약
- 섹션 4 "주의사항": 4순위 블로그 1개만 ⚠️ 부족

판정: 블로커 (섹션 3 빈약)

권장 액션:
- blog-research 재호출로 "실전 사례" 섹션용 자료 보강
- 또는 섹션 3 을 기획에서 제외 (대신 섹션 2 에 통합)
```

### D2 — 복잡도 판단

**검사 방법**:

1. 기획안에 명시된 모드 확인 (단편 / 시리즈 N편)
2. 자료 양과 모드의 일치 확인:
   - 단편인데 자료가 너무 많음 (1순위 5개+, 2순위 5개+) → 시리즈 권장 가능
   - 시리즈인데 자료가 부족 (편당 1순위 1개 미만) → 단편 권장 가능
   - 단편이고 자료 적당 (1순위 2~5개) → 통과
   - 시리즈고 자료 충분 (편당 1순위 2개+) → 통과
3. 주제 자체의 폭 평가:
   - 주제가 한 개념인데 시리즈로 잡힘 → 단편 권장
   - 주제가 여러 하위 개념을 포함하는데 단편으로 잡힘 → 시리즈 권장 검토

**판정**:

- 자료-모드 미스매치 → **경고** (블로커는 아님 — writer 가 어느 정도 처리 가능)
- 일치 → 통과

**예시**:

```
[D2 복잡도 판단]

기획안 모드: 단편
자료 양: 1순위 6개, 2순위 4개, 4순위 2개

⚠️ 경고: 단편치고 자료가 많음. 시리즈 (2~3편) 로 전환하면 각 편이 더 깊어질 수 있음.

권장 액션:
- 시리즈 전환 검토 (사용자 확인 필요)
- 또는 단편 유지하되 자료 일부를 "더 알아보기" 외부 링크로 정리
```

### D3 — 제목/설명 품질

**검사 방법**:

1. 기획안의 제목 후보 개수 확인:
   - 0개 → **블로커** (제목 자체가 빠짐)
   - 1개 → **경고** (선택지 부족)
   - 2~3개 → 통과
2. 각 제목 후보를 §META-TITLE 기준으로 평가:
   - 한국어 20자 이내?
   - 콜론 구조? (`:` 포함된 제목은 §RULE-COLON-HEADING 위반)
   - 호기심 유발? (의문형, 역설, 구체 시나리오)
   - 군더더기 없음?
   - 약어/전문용어가 전면에 주어로 배치됐는가? §META-TITLE 허용 조건(핵심 기술 식별자 / 대중 약어 / 대상 독자가 통상 아는 생태계 약어)에 해당하지 않으면 위반. 위반 시 현상 중심 풀어쓰기 대안 제안
3. 설명 (50자 내외) 평가:
   - 존재 여부
   - 길이 (50자 내외인가)
   - 존댓말?
   - 막히는 지점이 구체적?
   - 약어/전문용어가 전면에 배치됐는가? §META-DESCRIPTION 동일 원칙 적용 (대상 독자 기준). 대중 독자면 풀어 쓰기 대안 제안

**판정**:

- 제목 후보 모두 §META-TITLE 위반 → 블로커
- 일부 위반 → 경고
- 설명 누락 → 경고
- 모두 통과 → 통과

**예시**:

```
[D3 제목/설명 품질]

제목 후보:
1. "useMemo: 언제 쓰고 언제 피할까" — ❌ 콜론 구조 (§RULE-COLON-HEADING 위반)
2. "useMemo, 측정하고 쓰세요" — ⚠️ 약함 (호기심 유발 부족)
3. "useMemo 가 느려질 때" — ✅ 호기심 유발

설명: "useMemo 의 동작 원리와 사용 가이드입니다."
- 50자 OK
- 존댓말 OK
- 막히는 지점: ⚠️ 추상적. "어디서 막히는지" 가 안 보임.

판정: 경고

권장 액션:
- 후보 1 (콜론 구조) 제거
- 새 후보 추가 생성 권장 (호기심 유발 + 콜론 없음)
- 설명 재작성 권장: "왜 useMemo 가 오히려 느려지는지, 측정 후 쓰는 법까지요."
```

### D4 — 섹션 구조

**검사 방법**:

1. H2 섹션 개수 확인:
   - 0~1개 → 블로커 (구조 자체가 없음)
   - 2개 → 경고 (너무 적음)
   - 3~6개 → 통과
   - 7개 이상 → 경고 (너무 많음, 시리즈 검토)
2. 각 섹션의 역할 명시 확인:
   - 모든 섹션이 역할 한 줄 명시? → 통과
   - 일부 누락 → 경고
3. 섹션 흐름 평가:
   - 도입 (문제 제기) → 본론 (정의/동작/사례) → 마무리 흐름?
   - 갑작스러운 주제 점프?
4. 빈약한 섹션 확인 (D1 와 연동):
   - 자료 0개 매핑된 섹션 → 빈약, D1 에서 이미 잡혔을 것
5. 중복 섹션 확인:
   - 같은 내용을 두 섹션에서 다루지 않는가?

**판정**:

- 섹션 0~1개, 또는 흐름 깨짐 → 블로커
- 일부 결함 → 경고
- 통과

**예시**:

```
[D4 섹션 구조]

섹션 개수: 5개 ✅
역할 명시: 모두 명시 ✅
흐름:
  1. useMemo 가 뭔가 (정의)
  2. 언제 도움이 되나 (효과)
  3. 언제 해롭나 (역효과)
  4. 측정 방법 (가이드)
  5. 마무리

✅ 통과 — 자연스러운 흐름

[취향] 섹션 4 "측정 방법" 이 약간 짧을 수 있음. 섹션 3 의 "역효과" 와 통합해서
"역효과를 측정하는 법" 으로 합치면 더 단단할 듯. 하지만 현재 구조도 충분.
```

### D5 — 내부 링크 후보 검증

**검사 방법**:

1. Phase 1 에서 찾은 내부 링크 후보 목록 확인
2. 각 후보의 배치 힌트 확인:
   - 어느 섹션에 들어갈지 명시?
   - 그 섹션과 의미적으로 연결되나?
3. 후보 자체의 신뢰성:
   - 실제 파일 경로 매칭? (글러브로 한 번 더 확인)
   - 같은 주제의 글이라 중복 위험?

**판정**:

- 후보가 0개여도 통과 (내부 링크는 옵션)
- 후보 있는데 배치 힌트 없음 → 경고
- 후보 매핑이 부자연스러움 → 경고

**예시**:

```
[D5 내부 링크 후보 검증]

후보 2개:
1. /posts/css-stacking-context (배치: 섹션 1 도입부)
   - 파일 실존 확인 ✅
   - 의미 매칭: useMemo 와 stacking context 의 연결이 약함 ⚠️
   - 평가: 경고. 정말 자연스럽게 배치 가능한지 writer 에게 위임할지, 아니면 제외할지

2. /posts/react-rendering-basics (배치: 섹션 2 본론)
   - 파일 실존 확인 ✅
   - 의미 매칭: useMemo 의 전제 개념으로 자연스러움 ✅
   - 평가: 통과

판정: 경고 (후보 1 의미 매칭 약함)

권장 액션:
- 후보 1 제외 또는 writer 에게 "억지로 넣지 말 것" 강조
```

## Step 3: 블로커/경고/취향 분류

D1~D5 결과를 종합해서 분류:

```
블로커: <개수>건
- D1 자료 충분성: 섹션 3 빈약

경고: <개수>건
- D2 복잡도: 단편치고 자료 많음
- D3 제목: 후보 1 콜론 구조 위반
- D5 내부 링크: 후보 1 의미 매칭 약함

취향: <개수>건
- D4 섹션 4 "측정 방법" 짧음, 섹션 3 과 통합 검토
```

## Step 4: 자동 수정 액션 생성

오케스트레이터가 직접 처리할 수 있는 액션을 구조화해서 반환.

### 자동 수정 액션 카테고리

**카테고리 1: blog-research 재호출**

- D1 자료 부족 시
- 액션: "topic=<원래 주제>, additional_topics=<부족한 섹션 키워드>, depth=basic"

**카테고리 2: 복잡도 전환**

- D2 미스매치 시
- 액션: "current_mode=single, suggested_mode=series, parts=2~3"

**카테고리 3: 제목 후보 추가 생성**

- D3 후보 부족 시
- 액션: "current_titles=[...], generate_more=N"

**카테고리 4: 내부 링크 재검색**

- D5 후보 의심 시
- 액션: "topic=<주제>, exclude=[현재 후보들], rescan=true"

### 액션 반환 형식

```markdown
## 자동 수정 액션 (오케스트레이터 처리)

### 액션 1: blog-research 보강 호출

이유: D1 — 섹션 3 "실전 사례" 자료 0개
파라미터:
topic: "useMemo 실전 사례"
user_urls: []
depth: basic

### 액션 2: 제목 후보 추가 생성

이유: D3 — 후보 1 (콜론 구조) 제거 후 후보 2개만 남음
파라미터:
current_titles: ["useMemo, 측정하고 쓰세요", "useMemo 가 느려질 때"]
generate_count: 2
```

오케스트레이터는 이 액션들을 순차 실행한 후 기획안을 갱신, draft-review 를 다시
호출하거나 GATE 1 로 진입.

## Step 5: 사용자 확인 항목 구조화

오케스트레이터가 자동 수정할 수 없는 항목 (큰 구조 변경, 주제 재해석) 만 사용자
확인 카테고리로.

### 반환 형식

```markdown
## 사용자 확인 필요 항목

### 항목 1: [D2 복잡도 전환]

문제:
단편으로 기획됐는데 자료가 1순위 6개 + 2순위 4개로 많음. 단편에 다 담기 어려움.

수정 옵션:

A) 시리즈 (2편) 으로 전환

- 1편: useMemo 의 동작과 효과
- 2편: useMemo 의 역효과와 측정

B) 단편 유지하되 자료 일부를 "더 알아보기" 외부 링크로 정리

- 본문에 핵심 자료 4개만, 나머지는 References 에 추가

C) 주제 축소 (단편 유지)

- 효과 측정 부분만 단편으로, "역효과" 만 다음 글로

기본값: A (시리즈 전환)

### 항목 2: [D4 섹션 통합 제안] [취향]

문제:
섹션 4 "측정 방법" 이 짧을 수 있음. 섹션 3 "역효과" 와 통합 검토.

수정 옵션:

A) 통합: 섹션 3 을 "역효과와 측정" 으로 확장
B) 분리 유지 (현재 기획)

기본값: B (현재 기획 유지 — 취향 차원)
```

## Step 6: 리포트 생성

```markdown
# draft-review 결과: <topic>

## 검사 항목

- D1 자료 충분성
- D2 복잡도 판단
- D3 제목/설명 품질
- D4 섹션 구조
- D5 내부 링크 후보 검증

## 발견된 항목

### 블로커 (1건)

- D1 섹션 3 "실전 사례" 자료 0개

### 경고 (3건)

- D2 단편치고 자료 많음 (시리즈 검토)
- D3 후보 1 콜론 구조
- D5 후보 1 의미 매칭 약함

### 취향 (1건)

- D4 섹션 4 통합 검토

### 통과

- D3 설명 품질 (재작성 권장 외)
- D5 후보 2 의미 매칭

## 자동 수정 액션 (오케스트레이터 처리)

(Step 4 결과)

## 사용자 확인 필요 (오케스트레이터가 AskUserQuestion 으로 처리)

(Step 5 결과)

## 종합 판정

블로커 1건 + 경고 3건. 자동 수정 액션 처리 후 GATE 1 진입 가능.
사용자 확인 항목은 GATE 1 전 또는 후에 처리.
```

## 제약

- **검사 대상은 기획안만**. MDX 파일 검사는 validator/expression/coherence 영역.
- **자동 수정은 오케스트레이터에게 액션 반환만**. draft-review 가 직접 Edit 하지 않음
  (기획안은 오케스트레이터 메모리에 있어서 파일 Edit 자체가 의미 없음).
- **신중함 원칙 준수**. 블로커는 강하게, 경고는 알리되 진행 허용, 취향은 가볍게.
- **너무 까다로우면 안 됨**. "writer 가 좋은 글을 쓸 수 있는가?" 만 판단.
- **글쓴이 의도 추정**. 시리즈 → 단편 같은 큰 결정은 사용자가 의도한 것일 수 있음.
- **SHARED.md 규칙을 SKILL.md 에 복사하지 마세요**. Read 로 참조만.
