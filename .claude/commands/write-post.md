# 입력

원본 인자: $ARGUMENTS

입력에서 **주제**와 **참고 자료**를 분리하세요:

- 공백으로 구분된 토큰 중 `http://` / `https://`로 시작하는 것은 **참고 URL**로 간주
- 나머지 텍스트가 **주제**
- 참고 URL이 있으면 `WebFetch`로 각 URL을 읽어 핵심 논점·예시·용어를 파악한 뒤 기획/집필에 반영 (단순 번역 금지, 관점과 예시는 자기 것으로 재구성)

## 자료 수집 우선순위 (1순위 = 공식 문서/원출처)

주제를 확정하기 전에 **반드시 권위 있는 출처를 먼저 찾으세요**. 참고 URL이 주어지지 않았어도 아래 순위대로 자료를 직접 수집합니다.

1. **공식 문서 / 명세 / 표준** - MDN, W3C/WHATWG 스펙, TC39, RFC, 언어·런타임 공식 문서(ECMAScript, Node.js, Python docs 등), 라이브러리/프레임워크의 공식 문서 및 공식 GitHub 저장소(README·CHANGELOG·이슈). **라이브러리/프레임워크 관련 주제는 `context7` MCP(`mcp__plugin_context7_context7__resolve-library-id` → `query-docs`)를 먼저 호출해 최신 공식 문서를 확인하세요.**
2. **원저자/핵심 구현체 글** - 스펙 저자, 프레임워크 메인테이너, 브라우저 엔진 개발자 블로그(v8.dev, webkit.org, web.dev 중 공식 포스트) 등 1차 출처
3. **검증 가능한 2차 자료** - 잘 알려진 엔지니어링 블로그, 컨퍼런스 발표, 논문. 단, 주장과 코드 예시는 반드시 1순위 자료와 교차 확인
4. **개인 블로그/커뮤니티 글(Medium, dev.to, Qiita, 티스토리 등)** - 최후 순위. 개념 설명용 보조 자료로만 쓰고, 그 안의 사실 주장을 글에 옮길 때는 1순위 출처로 재검증

규칙:

- 1순위 자료를 찾지 못한 상태로 글을 쓰지 마세요. 최소 한 개의 공식 문서/스펙/공식 저장소 링크가 확보되어야 기획 단계로 넘어갑니다
- 사용자가 개인 블로그 URL만 참고로 던졌더라도, 그 글에서 인용한 **원출처(스펙/공식 문서)를 역추적해 직접 확인**하세요
- 정확한 버전·브라우저 호환성·API 시그니처는 항상 공식 문서 기준으로 기술. 기억이나 2차 자료로 쓰지 말 것

# 작업 흐름

반드시 아래 순서로 진행하세요.

## 0단계. 이전 글 스캔 (내부 링크 후보 수집)

기획안을 만들기 전에 **이 블로그에 이미 발행된 글 중 관련 있는 것**을 먼저 찾아두세요. 독자가 새 글을 읽다가 "이 부분은 저 글에서 더 깊게 설명했었지"로 자연스럽게 넘어갈 수 있어야 합니다.

절차:

1. `content/posts/**/*.mdx`를 훑어 현재 주제와 겹치거나 선행 지식이 되는 글을 찾습니다. 각 파일의 frontmatter(`title`, `description`, `tags`, `series`)와 H2 섹션만 빠르게 확인하면 충분합니다. Grep/Glob으로 핵심 키워드·태그를 먼저 거르고, 후보만 Read로 구조 확인.
2. 후보를 3~5개 이내로 추립니다. 기준:
   - 새 글의 **선행 개념**을 설명한 글 (예: `position` 설명 글을 쓰는데 이미 `stacking context` 글이 있음)
   - 같은 **시리즈에 속할 만한 글** (동일 `series` 문자열이면 새 글도 해당 시리즈 편으로 편입 검토)
   - 같은 **태그**를 공유하고 주제가 맞닿은 글
   - 새 글에서 "자세한 건 이 글에서"로 떼어낼 수 있는 **깊은 주제**를 다룬 글
3. 기획안의 `참고 자료` 아래에 **"내부 링크 후보"** 항목으로 명시하세요. URL은 단편이면 `/posts/<slug>`, 시리즈 편이면 `/series/<seriesSlug>` 또는 `/posts/<partSlug>` 경로로 적습니다.
4. 후보가 하나도 없으면 "내부 링크 후보: 없음"이라고 적으면 됩니다. 억지로 연결하지 마세요.

기획안 `참고 자료` 섹션 예시:

```
참고 자료:
  외부: https://developer.mozilla.org/...
  내부 링크 후보:
    - /posts/css-stacking-context - "쌓임 맥락" 섹션에서 선행 개념으로 연결
    - /posts/transform-vs-top-left-rendering-pipeline - 렌더링 파이프라인 이야기할 때 참고
```

## 1단계. 복잡도 판단 → 기획안 제시

주제의 깊이·범위를 보고 **단편** 또는 **시리즈(2~5편)** 중 하나를 고릅니다.

- **단편**: 하나의 개념/문제/패턴을 1편에 담을 수 있을 때. 범위가 좁거나, 실용 팁 묶음, 특정 API/속성 설명, 짧은 회고 등
- **시리즈(2~5편)**: 여러 하위 개념이 순차적으로 심화되거나, 한 편에 담으면 3000자를 크게 초과할 때. 단순하면 2편, 깊이가 필요하면 최대 5편

참고 URL이 있다면 그 내용의 분량/주제 범위도 복잡도 판단에 반영하세요.

### 단편 기획안 형식

```
📋 단편 기획안: <글 제목>

slug: <kebab-case-영문>
예상 분량: <1500~3000자 범위> (예상 읽기 ~N분)

핵심 메시지: <한 줄>
주요 섹션:
  - <섹션1>
  - <섹션2>
  - ...
참고 자료: <URL 목록 또는 "없음">

이대로 진행할까요? (수정 요청 / "진행" / "취소")
```

### 시리즈 기획안 형식

```
📋 시리즈 기획안: <시리즈 제목>

시리즈 slug: <kebab-case-영문>
편수: <N>편

1. <편 제목>
   설명: <한 줄>
   핵심 주제: <쉼표로 구분>
2. ...

참고 자료: <URL 목록 또는 "없음">

이대로 진행할까요? (수정 요청 / "진행" / "취소")
```

- slug는 **영문 kebab-case**
- **제목(title)**: 한국어, **20자 이내**, 클릭하고 싶게. 설명형("~에 대한 정리")보다 질문·호기심 유발형("왜 ~할까", "~하는 이유")을 우선. 키워드는 가능한 드러내되 군더더기("이야기", "살펴보기", "알아보기")는 뺀다
- **설명(description)**: **50자 내외**, 독자가 겪는 장면이나 막히는 지점을 구체적으로. 본문 요약이 아니라 "이 글이 풀어주는 문제"를 한 문장으로
- 시리즈인 경우 각 편은 독립적으로 읽히되 순서대로 읽으면 심화
- 사용자가 수정 요청하면 반영 후 다시 확인
- 사용자가 "진행" / "ㅇㅇ" / "가자" 같은 승인 표현을 주기 전까지는 2단계로 넘어가지 마세요

## 2단계. 작성 및 저장

승인 후 글을 작성해 저장합니다.

- **단편**: `content/posts/<slug>.mdx` 한 파일로 저장
- **시리즈**: `content/posts/<seriesSlug>/<partSlug>.mdx`에 한 편씩 차례대로 저장. 시리즈 디렉토리가 없으면 만드세요. 한 편 쓰고 바로 다음 편으로 넘어가세요 (진행 상황은 간단히 한 줄씩 알려주세요)

### Frontmatter 스키마 (velite.config.ts 기준)

**시리즈 편**:

```yaml
---
title: "<편 제목>"
description: "<한 문장 설명, 50자 내외>"
date: <YYYY-MM-DD, 오늘 날짜>
tags: ["<태그1>", "<태그2>", "<태그3>"]
series: "<시리즈 제목 - 모든 편이 정확히 동일한 문자열이어야 함>"
seriesOrder: <1부터 시작하는 숫자>
published: true
---
```

**단편**: `series` / `seriesOrder` 필드는 생략.

```yaml
---
title: "<글 제목>"
description: "<한 문장 설명, 50자 내외>"
date: <YYYY-MM-DD, 오늘 날짜>
tags: ["<태그1>", "<태그2>", "<태그3>"]
published: true
---
```

- `series` 문자열은 **모든 편이 완전히 동일해야** 그룹화됩니다. 오타/공백/접미사 차이 금지.
- 시리즈의 `tags`는 편끼리 통일.
- `date`는 오늘 날짜 (`date` 명령 또는 컨텍스트의 currentDate 참조).

### 사용 가능한 MDX 컴포넌트

```jsx
<Callout variant="tip|warning|info|error">
내용
</Callout>

<AnimatedStep steps={[
  { title: "제목", content: "설명", code: "코드(선택)" },
  { title: "제목", content: "설명" }
]} />

<CodePlayground
  code={`코드 내용`}
  css={`선택: 별도 CSS 파일`}
  template="react"
  showPreview={true}
/>

<VideoEmbed src="https://youtube.com/watch?v=..." title="제목" />
```

**CodePlayground 주의사항**:

- `template="react"`: 코드는 `function 컴포넌트() { return <div>...</div> }` 형태. `export default`는 자동 주입되므로 작성하지 않아도 됨. 단 함수명은 **PascalCase**로.
- `template="vanilla"`: CSS 스니펫은 바로 쓰면 `/styles.css`로 자동 분리됨. JS가 필요하면 `vanilla` 대신 `react`를 권장.
- `code`는 반드시 백틱 템플릿 리터럴(`` ` ``)로 감싸고, 내부에 백틱 쓸 일 있으면 이스케이프.

**인라인 스타일 금지 - `css` prop으로 분리**:

React 예제에서 `style={{ ... }}` 인라인 스타일을 여러 줄 쓰면 독자가 로직을 읽기 어렵습니다. 스타일이 있으면 **무조건** `css` prop으로 별도 CSS 파일로 뽑으세요.

- 스타일 속성이 2~3개만 있어도 className으로 빼고 `css` prop에 정의 (컴포넌트 코드는 구조와 로직만 드러나게)
- `css` prop이 넘어오면 `styles.css`가 자동으로 생성되고 `App.js`에 `import "./styles.css";`가 주입됩니다. 편집기 탭에도 같이 노출되니 독자가 바로 볼 수 있음
- 여러 변형(state, variant)은 BEM 스타일 modifier 클래스(`.box--bad`, `.sample--nowrap`)로 표현
- 절대로 `style={{ ... }}`를 예제 내부에 남기지 마세요. 스타일 분리가 곧 가독성입니다

```jsx
// ❌ 인라인 스타일 덩어리 - 예제 코드가 스타일에 파묻힘
<CodePlayground
  code={`function Demo() {
  return <div style={{ width: 200, border: "1px solid #ccc", padding: 10 }}>...</div>;
}`}
  template="react"
/>

// ✅ css prop으로 분리 - 컴포넌트는 구조만, 스타일은 CSS 파일에
<CodePlayground
  code={`function Demo() {
  return <div className="box">...</div>;
}`}
  css={`.box {
  width: 200px;
  border: 1px solid #ccc;
  padding: 10px;
}`}
  template="react"
/>
```

**JSX prop 문자열 규칙 - 여러 줄은 무조건 백틱**:

`<CodePlayground>`의 `code`/`css`, `<AnimatedStep>`의 `code` 필드처럼 여러 줄 문자열을 JSX prop으로 넘길 때 지켜야 할 것들:

1. **큰따옴표(`"..."`) 금지**. 속성값에 실제 개행을 넣으면 렌더가 깨집니다. 여러 줄이 필요하면 중괄호 + 백틱(`{\`...\`}`)으로 감싸세요. `<Callout>` 자식 텍스트는 마크다운이라 여러 줄 허용이지만, **속성값**에 여러 줄이 필요하면 예외 없이 백틱입니다.
2. **들여쓰기는 템플릿 리터럴 내부 기준 0칸부터**. 독자에게 있는 그대로 렌더되므로 JSX 들여쓰기에 끌려가면 안 됩니다.
3. **중첩 백틱 안의 줄바꿈은 `\n`으로**. 실제 개행을 넣으면 MDX 포매터가 들여쓰기를 재조정하면서 코드가 깨집니다. 바깥 템플릿 리터럴 자체의 줄바꿈은 그대로 써도 돼요 - 문제가 되는 건 **중첩 백틱** 안입니다.
4. `{`, `}`, `` ` ``, `$` 같은 특수문자가 값 안에 들어가면 이스케이프 필수.

```jsx
// ❌ 큰따옴표 + 개행 - 렌더 깨짐
<CodePlayground code="const a = 1;
const b = 2;" />

// ❌ 중첩 백틱 안에 실제 개행 - 포매터가 재정렬하면서 깨짐
code={`function Demo() {
  const text = \`여러    공백과
줄바꿈이
있는    텍스트\`;
}`}

// ✅ 중괄호+백틱, 내부 들여쓰기 0칸부터, 중첩 백틱 안은 \n
<CodePlayground
  code={`function Demo() {
  const text = \`여러    공백과\n줄바꿈이\n있는    텍스트\`;
}`}
/>
```

### 예상 읽기 시간

분당 500자 기준으로 자동 계산되어 메인 카드와 상세 헤더에 붙습니다. **frontmatter에 분을 따로 쓰지 마세요** - 본문 길이만 맞추면 됩니다. 감 잡을 수 있게 대응표: **1500자 ≈ 3분 / 2500자 ≈ 5분 / 3500자 ≈ 7분**. 기획안에는 예상 분량 옆에 "(예상 읽기 ~N분)"을 같이 적어 독자가 호흡을 예측할 수 있게 하세요. 1분이면 너무 얕고 10분을 넘으면 과하게 길다는 신호 - 둘 다 기획을 다시 잡으세요.

### 글 작성 규칙

- **분량**: 1편당 1500~3000자 (한글 기준). 너무 짧으면 얕고, 너무 길면 지루함
- **오프닝**: 독자가 마주하는 실제 문제로 시작. 이론 나열 금지
- **구조**: H2로 섹션 나누고, 필요하면 H3로 하위 세분화
- **시각화**: 개념은 `<AnimatedStep>`으로 단계 분해
- **실행 가능한 예제**: 핵심 개념마다 `<CodePlayground>`
- **팁/주의사항**: `<Callout>`
- **마무리**:
  - 시리즈 중간 편 → 다음 편 예고
  - 시리즈 마지막 편 → 시리즈 전체 회고
  - 단편 → 핵심 요약 + 한 걸음 더 나아갈 힌트
- **내부 링크 (이전 글 자연스럽게 연결)**: 0단계에서 추려둔 후보 글을 본문에 자연스럽게 녹이세요. 원칙:
  - **문장 안에 녹이기**. 별도의 "관련 글" 박스를 만들지 말고, 개념이 처음 등장하는 자리에서 바로 `[쌓임 맥락](/posts/css-stacking-context)`처럼 링크. "자세한 건 [이 글](/posts/...)에서"가 아니라 "쌓임 맥락이 어떻게 만들어지는지는 [지난 글](/posts/css-stacking-context)에서 다뤘어요"처럼 **링크 앞뒤로 문맥**을 붙이세요.
  - **경로는 반드시 절대경로 `/posts/<slug>` 또는 `/series/<seriesSlug>`**. 이 두 가지 외의 형태는 전부 금지:
    - ❌ `/blog/<slug>`, `/post/<slug>`, `/articles/<slug>` 같은 임의 접두사 (이 블로그 라우트는 `app/posts/[slug]` 뿐)
    - ❌ `.mdx` 확장자 포함, `http(s)://` 풀 URL, 상대경로 (`./`, `../`)
    - ❌ 파일 경로 그대로(`content/posts/...`)
    - `<slug>`는 **실제 파일명**과 정확히 일치해야 합니다. `content/posts/<slug>.mdx` 또는 시리즈 편이면 `content/posts/<seriesSlug>/<partSlug>.mdx`의 파일명을 그대로 옮겨쓰세요. 기억이나 추측으로 적지 말고, 0단계에서 스캔할 때 본 실제 파일명을 복사해 쓰세요.
  - **한 글에 2~4개 정도가 적당**. 남발하면 독자가 본문을 못 읽고 이탈합니다. 딱 필요한 자리에만.
  - **선행 개념 → 본문 초반, 심화 주제 → 본문 후반이나 마무리** 위치가 자연스럽습니다. 마무리 "한 걸음 더" 힌트에 1~2개 엮는 것도 좋음.
  - **후보가 없거나 억지스러우면 과감히 생략**. 억지 링크는 안 거는 게 낫습니다.
  - 내부 링크는 본문용이고, 하단 `<References />` 섹션의 외부 출처와는 별개입니다. `<References />`에 내부 글을 넣지 마세요.
- **참고 자료 (출처 표기, 모든 글에 필수)**: 본문 맨 아래에 **`<References />` MDX 컴포넌트**를 **항상** 두고, 글을 쓰며 실제로 참고한 모든 출처를 나열하세요. 입력 URL이 없고 스스로 조사한 경우에도 동일하게 출처를 남깁니다. `## 참고` 같은 마크다운 섹션 헤딩은 쓰지 마세요 - 컴포넌트가 자체 헤딩을 렌더합니다.
  - 형식: `<References items={[...]} />`. 각 항목은 `{ id, title, href, description }` 객체. `id`는 kebab-case 짧은 식별자(예: `mdn-word-break`, `web-dev-rendering`)로, 본문 `<Cite>`와 연결하는 앵커 용도. `title`과 `description` 모두 em-dash(`—`)가 아니라 짧은 하이픈(`-`).
  - **1순위 출처(공식 문서·스펙)를 배열 맨 위에 배치**하고, 2차/개인 블로그는 아래로. 최소 하나의 공식 출처가 포함되어야 합니다
  - `description`에는 "어떤 부분을 참고했는지" 한 줄로 간단히 덧붙이세요
  - 접근 가능한 안정적인 URL만 사용 - 로그인/유료 장벽이 있는 링크는 피하고, 가능하면 캐노니컬 주소(예: `developer.mozilla.org/en-US/docs/...`)
- **본문 인용 마크 `<Cite id="...">`**: 본문에서 특정 주장/인용 옆에 하단 참고자료로 점프하는 작은 인포 마크를 달 수 있습니다. `<References>`의 `id`와 동일한 값을 `<Cite id="...">`로 넘기면, 클릭 시 해당 항목으로 스크롤되고 하이라이트됩니다.
  - **문단(혹은 한 줄)마다 최대 1개**만 두세요. 같은 문단에 여러 개가 붙으면 산만해집니다. 여러 출처를 참고한 문단이라면 가장 핵심이 되는 하나만 `<Cite>`로 표시하고, 나머지는 `<References>` 리스트로만 남기면 됩니다
  - **텍스트 바로 뒤에 공백 없이** 붙여야 줄바꿈 경계가 깨지지 않음: `...지원됩니다.<Cite id="mdn-xxx" />`
  - Cite는 선택 사항. 본문에 전혀 쓰지 않아도 되고, 핵심 주장 한두 군데에만 붙여도 충분합니다
- **영어 원문 인용은 항상 한글로 풀어쓰기**: 스펙·공식 문서·원저자 글에서 영어 문장을 인용할 때는 반드시 **원문 바로 아래에 한국어로 이어받는 해석**을 붙이세요. 독자가 영어 문장만 보고 넘어가지 않게 하기 위함입니다.
  - **형식은 `<Callout variant="warning">`**(또는 `info`)을 쓰고, 첫 줄에 `"원문 문장." - 출처`, 빈 줄 하나, 다음 줄에 글쓴이 목소리로 풀어쓴 한국어 문장. 출처 구분자는 em-dash(`—`)가 아니라 짧은 하이픈(`-`).
  - **축자 번역 금지**. "(번역) ..."처럼 기계적으로 옮기지 말고, 인용의 **요지를 본문 톤으로 이어받아** 풀어 쓰세요. 독자가 원문 → 해석을 한 호흡으로 읽을 수 있어야 합니다. 해석 문장 안에 원문의 핵심 표현(예: `"Flex의 유연성을 꺼서"`)을 따옴표로 살짝 되살려주면 원문과 리듬이 맞습니다.
  - **용어는 원문 그대로**(`flex-item`, `stacking context` 등). 한두 단어짜리 짧은 용어는 이 규칙 적용 대상이 아니고, **문장 단위 인용**에만 적용합니다.

  ```mdx
  <Callout variant="warning">
  "If you are using flexbox and find yourself disabling some of the flexibility, you probably need to use CSS grid layout." - MDN

  flex-item에 고정 width를 박고 있다면, 그건 "Flex의 유연성을 꺼서" 2D를 흉내 내고 있는 겁니다. Grid 한 줄이면 끝날 일이에요.

  </Callout>
  ```

- 독자가 "아하!" 하는 순간을 만드세요

### 사람처럼 쓰기 (AI 티 내지 않기)

설명서가 아니라 동료가 카페에서 얘기해주는 느낌으로. 구조는 명확하되 문장은 사람 냄새가 나게.

**하지 말 것**

- **콜론(`:`) 정의식 남용**: "핵심은: ...", "이유: ..." 처럼 문장마다 반복하면 기계 같음. 문장 안에 자연스럽게 흘리거나 H3/목록으로 구조 분리
- **em-dash(`—`) 금지**: 본문 전체에서 em-dash(`—`)를 쓰지 말고 **짧은 하이픈(`-`)** 을 쓰세요. 삽입구·부연 설명·출처 표시·링크 설명 구분자 등 기존에 `—`를 쓰던 자리 모두 `-`로 통일. 부연이 길면 `-` 대신 쉼표·마침표·괄호로 끊는 게 더 자연스럽습니다
- **병렬 삼단 구조**: "빠르고, 안전하고, 쉽습니다" 같은 3단 나열을 섹션마다 반복하면 AI 티가 남
- **과장 형용사**: "완벽한", "강력한", "혁신적인", "놀라운" 같은 마케팅 단어는 구체적인 동작·수치·예시로 대체
- **메타 문장**: "이번 글에서는 ~를 다뤄보겠습니다", "살펴보겠습니다", "결론적으로 말씀드리면" 같은 예고·정리 문장. 바로 본론으로

**해야 할 것**

- **어투 섞기**: "~입니다"와 "~죠/~네요/~거든요"를 자연스럽게 섞으세요. 모든 문장이 "~합니다"로 끝나면 기계 같음. 단 반말은 금지
- **호흡 다양화**: 짧은 문장과 긴 문장을 섞어야 리듬이 삽니다
- **자기 목소리**: 개발하면서 겪는 혼잣말·의문·시행착오를 드러내세요. "처음엔 이게 왜 안 되나 싶었어요" 같은 한 줄이 독자 경험과 맞닿습니다

### 마크다운 리스트 주의 - 마커가 렌더되지 않음

이 블로그의 `.prose` 스타일은 Tailwind preflight의 `list-style: none` 기본값을 그대로 쓰기 때문에 **마크다운 `- 항목`이나 `1. 항목`은 점·숫자 마커가 보이지 않고** 들여쓴 문단처럼만 렌더됩니다. `- 항목 A` 세 줄 나열하면 그냥 세 문단이 붙어 보여서 독자가 "이게 뭐지?" 싶어집니다.

그래서 **bare 마크다운 리스트 사용 금지**. 대신:

- 항목이 2~4개고 짧으면 → **문단으로 풀어쓰기**. "첫째는 X, 둘째는 Y, 셋째는 Z입니다" 식으로 자연스럽게 이어주세요.
- 단계/흐름이 중요하면 → `<AnimatedStep>`으로 구조화 (각 스텝에 title/content/code).
- 팁·경고·체크리스트면 → `<Callout>` 안에 넣거나, 리스트 마커를 **문자 그대로 텍스트에 포함**(`• 항목`, `(1) 항목`, `- 항목`).
- 강조가 필요한 나열이면 `<strong>`이나 H3로 섹션 분리.

```
❌ - **Chrome**: 완벽 지원
   - **Safari**: 완벽 지원
   - **IE 11**: `keep-all` 미지원

✅ Chrome과 Safari는 전부 지원하지만, IE 11은 `keep-all`만 빠집니다.
✅ <Callout variant="info">
   • Chrome/Safari - 완벽 지원
   • IE 11 - keep-all 미지원
   </Callout>
```

굳이 번호 매긴 나열이 필요하면 마크다운 `1.` 대신 본문에 `1)`이나 `①`을 직접 써도 됩니다.

**체크리스트는 `<Callout>` + `☐` + `<br/>` 패턴으로**:

마무리 섹션의 "배포 전 체크리스트" 같은 나열은 항상 아래 포맷으로 쓰세요. `☐`만 줄줄이 적으면 CommonMark가 연속된 줄을 한 문단(= 한 줄)으로 합쳐서 시각적으로 다 붙어버립니다. 그래서 **각 항목 끝에 `<br/>`**, 그룹 헤딩 뒤에는 빈 줄을 둬야 실제로 줄바꿈이 보입니다.

```mdx
<Callout variant="info">
**반응형**

☐ 모든 화면 크기에서 텍스트가 적절히 줄바꿈되는가?<br/>
☐ 긴 URL이 레이아웃을 깨뜨리지 않는가?<br/>
☐ 모바일에서 가로 스크롤이 발생하지 않는가?

**접근성**

☐ 스크린 리더가 텍스트를 올바르게 읽는가?<br/>
☐ 200% 확대 시에도 가로 스크롤이 없는가?

</Callout>
```

규칙 요약: ① `- [ ]` 마크다운 task list는 마커가 렌더되지 않으니 **금지**, ② `☐`(U+2610)를 문자 그대로 사용, ③ 각 항목 끝에 `<br/>`(마지막 항목은 선택), ④ 그룹 헤딩(`**반응형**` 등) 아래는 빈 줄로 분리, ⑤ 체크리스트 앞에 "~을 정리했습니다" 같은 AI식 예고 문장 달지 말 것 - 짧은 한 문장이면 충분(예: "배포 전에 한 번씩 훑어볼 항목들입니다").

### 한국어 + 마크다운 함정 (반드시 준수)

CommonMark 강조(`**`) 닫는 규칙 때문에 한글 문장에서 깨지는 경우가 있습니다:

- ❌ `다음 편 **"word-break"**에서는` - 닫는 `**` 뒤가 한글이라 닫히지 않고 리터럴 노출
- ✅ `다음 편 **word-break** 에서는` - 닫는 `**` 뒤에 공백
- ✅ `다음 편인 **word-break** 를 살펴봅니다`

규칙: **`**`로 감싼 뒤에는 반드시 공백이나 구두점(`.`, `,`, `)`, `!`, `?`)을 둘 것.** 한글 음절이 바로 붙으면 안 됨. 인용부호(`"`)로 감싼 텍스트를 `\*\*`로 한 번 더 감싸지 말 것.

동일한 이유로 이탤릭 `*text*`도 한글 인접 시 주의. 불확실하면 `<strong>` 태그를 쓰세요.

## 3단계. 검증

각 파일을 저장한 직후 아래를 확인:

1. **Frontmatter**: 모든 필수 필드(`title`, `description`, `date`, `published`) 존재. 시리즈라면 `series` 문자열이 다른 편과 일치, 단편이라면 `series`/`seriesOrder` 미포함
2. **JSX 태그 균형**: `<Callout>`, `<AnimatedStep>`, `<CodePlayground>`, `<VideoEmbed>`가 짝이 맞거나 self-closing으로 닫혀있음
3. **마침 상태**: 본문이 마침표/물음표/느낌표/닫는 태그로 끝나야 함 (중간 잘림 방지)
4. **마크다운 강조**: 위 "한국어 + 마크다운 함정" 규칙 위반 없음
5. **출처 섹션**: 본문 맨 아래 `<References items={[...]} />` 컴포넌트가 존재하고, 최소 한 개 이상의 공식 문서/스펙 링크가 포함되어 있음. 없으면 다시 자료를 찾아 보강. `## 참고` 같은 마크다운 헤딩을 쓰지 않았는지도 확인. `<Cite id="...">`를 썼다면 모든 id가 `<References>`의 `id`와 1:1 매칭되는지, **한 문단에 Cite가 2개 이상 들어간 곳이 없는지** 점검
6. **내부 링크 유효성 (반드시 grep으로 기계 검증)**: 저장한 파일에서 아래 순서대로 확인:
   1. 먼저 잘못된 접두사/형식을 **Grep으로 전부 찾아냅니다**. 새로 쓴 mdx 파일 경로를 대상으로:
      - `\]\((?!/posts/|/series/|https?://|#)` - `/posts/`, `/series/`, 외부 URL, 앵커(`#`)가 아닌 모든 마크다운 링크 목적지를 잡아냄. 여기 걸리면 **100% 잘못된 경로**이므로 즉시 수정
      - `\]\(/blog/`, `\]\(/post/`, `\]\(/article` - 특히 자주 실수하는 접두사들. 이 블로그의 내부 라우트는 `/posts/<slug>`와 `/series/<seriesSlug>` **두 개뿐**이고 다른 접두사는 전부 404
      - `\]\([^)]*\.mdx\)` - `.mdx` 확장자가 링크에 들어가면 안 됨
   2. 걸린 링크가 없으면 다음으로, 본문에 남아있는 `/posts/<slug>` / `/series/<seriesSlug>` 링크 각각에 대해 **실제 파일이 존재하는지 Glob으로 확인**:
      - `/posts/<slug>` → `content/posts/<slug>.mdx` 또는 `content/posts/*/<slug>.mdx`가 실제로 존재해야 함
      - `/series/<seriesSlug>` → `content/posts/<seriesSlug>/` 디렉토리가 존재하고, 그 안 편들의 frontmatter `series` 문자열이 서로 일치해야 함
   3. 매칭 실패한 링크는 즉시 수정하거나, 대체할 수 없다면 해당 문장에서 링크를 **과감히 제거**. 깨진 링크를 남기느니 평문이 낫습니다.
7. **velite 스키마 검증**: 전부 작성 후 한 번 `npx velite` 실행해서 통과하는지 확인. 에러 나면 해당 파일 고치고 재실행.

문제 발견 시 해당 파일을 Edit으로 직접 고치세요. 재작성하지 말고 국소 수정 우선.

# 중요 원칙

- 사용자 승인 없이 기획안 단계를 건너뛰지 마세요
- 복잡도 판단은 사용자가 원하면 단편↔시리즈로 바꿀 수 있게 열려있게 제안
- 기존 글/시리즈 디렉토리를 덮어쓰지 마세요. 충돌 시 사용자에게 확인
- **자료 수집 1순위는 공식 문서/스펙/원저자 글**입니다. 라이브러리·프레임워크 주제면 `context7` MCP부터, 웹 표준/브라우저 API면 MDN·W3C·WHATWG부터 확인하세요. 개인 블로그는 보조 자료로만 쓰고, 그 안의 사실 주장은 1순위 출처로 재검증
- 참고 자료는 자기 언어로 재구성. 문단 복붙·번역 금지
- **모든 글은 본문 맨 아래 `<References items={[...]} />` 컴포넌트에 실제 참고한 출처를 남깁니다.** 1순위 공식 출처를 최소 하나 포함하고, 포맷은 "글 작성 규칙"의 예시를 따르세요. `<Cite id="...">`로 본문 인용 마크를 달 경우 한 문단에 하나까지만
- 오늘 날짜는 대화 컨텍스트의 `currentDate`를 사용 (없으면 `date +%Y-%m-%d`)
- 본문에 frontmatter를 중복 포함하지 마세요
- 파일 저장 후 반드시 검증. 검증 실패한 채로 다음 편 쓰지 말 것
